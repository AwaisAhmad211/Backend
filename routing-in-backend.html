HTTP Method = The “What”

  • GET → I want to fetch/read data
  • POST → I want to create something
  • PUT/PATCH → I want to update/replace
  • DELETE → I want to remove something

00:00:34.079and the role of routing is it expresses the where of a request — where do you want to send your intention / which resource you want to perform your action or your intent on

2. Routing = Mapping URL → Server-Side Logic

00:01:55.000to summarize it you can say that routing is basically mapping URL parameters to a server side logic and that's all there is to it

Routing in One Sentence

The combination of HTTP Method + Route/Path tells the server:

  • What action to perform (GET, POST, PUT…)
  • Where / on which resource to perform it (/users, /books, /orders/123…)

Server matches this combo → executes the correct handler → runs business logic, DB ops, etc. → returns response

4. Route Types & Patterns

Static Routes

00:04:00 approxroutes that are fixed → /api/users, /about, /login — no variables

Dynamic / Parameterized Routes

00:05:30 approxroutes with placeholders → /api/users/:id, /products/:slug

Nested Routes

00:06:00 approxroutes inside routes → /api/users/:userId/orders, /api/users/:userId/orders/:orderId

Catch-All / Wildcard Routes

00:23:00.279we are sending requests to a route which the server does not serve for right we are doing /api/v3/products at this point the server does not serve responses for this route right it does not have a Handler it does not cater to requests coming into this endpoint so typically what the server will do is in the end after serving all the different routes and all the methods uh associated with those route in the end it will do is /* whatever request after going through all the previous route matching algorithms whatever request reaches here this part it will be mapped to a Handler and that Handler will send a user-friendly message that this route which you are requesting this route does not exist right this route not found

Purpose of Catch-All Route

  • Catches any unmatched route
  • Returns 404 Not Found with friendly message
  • Prevents default ugly server errors
  • Usually placed as the very last route

API Versioning via Routes

00:20:01.679let's say you have an API endpoint and you are returning data in a particular format right and later on new requirements came in… your first option is changing the whole route right changing the whole route matching part so let's say you can say /api/new-products right or what you can do is you can add that to your versioning you can say in the version one of our API we were serving responses in this format and new requirements came in and in version two we are serving in this format

Benefits of Route Versioning (v1, v2…)

  • Clearly expresses version of response format
  • Allows breaking changes without breaking old clients
  • Gives clients a migration window
  • Eventually deprecate old versions (e.g., V1 → sunset)

5. Final Summary & Takeaways

Quick Mastery Checklist – Routing Fundamentals
  • HTTP Method = What (intent/action)
  • Route/Path = Where (resource/target)
  • Routing = Mapping Method + Path → Handler
  • Static routes vs Dynamic (:id) vs Nested vs Catch-all (/*)
  • Versioning via path (/v1/users vs /v2/users) → graceful deprecation
  • Always add catch-all route → return friendly 404

Understand this → reading/modifying any backend routing code becomes intuitive.
