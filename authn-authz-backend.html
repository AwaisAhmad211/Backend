1. Core Definitions – AuthN vs AuthZ

00:00:00.160and finally my favorite topic which is authentication and authorization and this is probably one of those areas which we encounter every day I'm pretty sure all of us have used authentication screens before you have we have logged into different platforms you have signed up into different platforms those are basically called the authentication flows

Two-Sentence Summary (Most Important Takeaway)

  • Authentication (AuthN) = Who are you? → Proving your identity (prove you are the user you claim to be)
  • Authorization (AuthZ) = What can you do? → After proving identity, what permissions/actions are you allowed to perform

Analogy: AuthN = showing your passport at airport security (who are you?). AuthZ = what boarding pass gates/areas you can access after passport check (what can you do?).

3. Modern Authentication Flows

Basic Username + Password Flow (Most Common)

01:30:00 approxmost basic form: user submits username + password → server hashes password → compares with stored hash → if match → authenticated

Correct Implementation Steps

  1. User submits username + password
  2. Server looks up user by username
  3. If user exists → hash submitted password with same salt/algorithm
  4. Compare hashed values → match = success, no match = fail
  5. Never store plain passwords → always use strong hashing (bcrypt, Argon2, PBKDF2)

Security Must-Know: Never Store Plaintext Passwords

Even if database is breached → attacker cannot use passwords directly

4. Timing Attacks – Real-World Vulnerability

01:32:42.480if the username is invalid in the first step the system will terminate there only and you will get uh some response that user not found and that will have a faster response time as compared to the system failing at the third step… attackers can find out whether that username was correct or not… using that parameter that delay… attackers can use this strategy… to further strategize their attack plan

Timing Attack Explained

  • Invalid username → immediate "User not found" (fast)
  • Valid username + wrong password → hash + compare → slower (extra work)
  • Attacker measures response time → infers valid usernames → brute-force only real users

Defenses Against Timing Attacks

MethodHow it WorksProsCons
Constant-time comparisonUse crypto-safe functions (no early exit on mismatch)Best securityRequires language support

Quick Mastery Checklist – AuthN & AuthZ Essentials
  • AuthN = Prove identity ("Who are you?")
  • AuthZ = Check permissions ("What can you do?")
  • Never store plain passwords → always hash + salt (bcrypt/Argon2)
  • Prevent timing attacks → constant-time compare or fake delay
  • Common models: RBAC (simple), ABAC (flexible), ReBAC (relationship-based)
  • Historical evolution: Trust/handshake → passwords → tokens → modern protocols
  • Always think security-first — small leaks become big breaches

Master these → you’ll build secure systems and confidently debug auth issues in any codebase.
