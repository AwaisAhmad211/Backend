Today’s Focus – HTTP Protocol

  • The medium through which browsers talk to servers
  • Send data (POST/PUT) or receive data (GET)
  • Most common way clients ↔ servers communicate

2. Two Foundational Ideas of HTTP

1. Statelessness (No Memory of Past Requests)

00:00:36.680the first one being statelessness what does statelessness basically mean? It has no memory of past interactions so each HTTP request carries all the necessary information for the server to process it such as headers or URLs and methods… and after the server responds it forgets about the request

Benefits of Statelessness

  • Simplicity — server doesn’t need to store session state → less complexity
  • Scalability — easy to distribute requests across many servers
  • Fault tolerance — server crash doesn’t lose client state

How do we simulate “state” in real apps?

00:02:13.040developers often Implement State management techniques like cookies, sessions, or tokens (JWTs) to maintain continuity in interactions where needed — like user logins or shopping carts

2. Client-Server Model

00:02:31.080in a typical HTTP request flow there is a client and there is a server always the client typically is a web browser or an application which initiates the communication by sending a request to the server… the server hosts resources like websites or apis or other content and waits for incoming requests

00:03:21.440HTTP protocol states that communication is always initiated by the client to get some kind of response by the server

3. HTTP Runs over TCP – Transport Basics

00:04:04.319to send some kind of request or receive some kind of response first the client and the server need to establish some kind of connection mechanism… for that HTTP uses TCP

TCP Guarantees HTTP Needs

  • Reliable delivery → no lost messages
  • Ordered packets
  • Error detection & retransmission
  • Connection-oriented (3-way handshake)

4. Anatomy of an HTTP Request

Four Main Parts

Part Example Purpose
Request Line GET /users HTTP/1.1 Method + Path + HTTP Version
Headers Host: api.example.com
Authorization: Bearer xyz
Accept: application/json
User-Agent: Mozilla/5.0…		 Metadata (client info, auth, content negotiation, caching…)
Empty Line (blank line) Separates headers from body
Body (optional) {"name":"John","age":30} Data being sent (POST/PUT/PATCH)

5. Anatomy of an HTTP Response

Part Example Meaning
Status Line HTTP/1.1 200 OK Version + Status Code + Reason Phrase
Headers Content-Type: application/json
Content-Length: 123
Cache-Control: max-age=3600		 Metadata about the response (type, size, caching…)
Empty Line (blank) Separates headers from body
Body {"id":1,"name":"John"} Actual payload (JSON, HTML, image, file…)

6. HTTP Methods – Semantics & Best Practices

Method Idempotent? Safe? Main Use Case Body Allowed? Common Status Codes
GET Yes Yes Retrieve resource (read-only) No 200, 404, 304
HEAD Yes Yes Get headers only (no body) No 200, 404
POST No No Create resource, submit data Yes 201, 400, 409
PUT Yes No Replace entire resource Yes 200, 204, 404
PATCH No (usually) No Partial update Yes 200, 204, 400
DELETE Yes No Remove resource Optional 204, 404
OPTIONS Yes Yes CORS preflight / allowed methods No 200, 204

7. Handling Large Files – Upload & Download

Uploading Large Files → Multipart Requests

01:12:00 approxwhenever we want to transfer large files to the server we should use multipart requests

Why Multipart?

  • Allows sending binary files + metadata together
  • Supports multiple files in one request
  • Standard way for file uploads (forms, APIs)

Downloading / Streaming Large Responses

01:13:35.480we are receiving chunks continuously from the server in different requests… important headers are:

  • Content-Type: text/event-stream → signals streaming
  • Connection: keep-alive → keep TCP connection open

This is how Server-Sent Events (SSE) or chunked transfer encoding works for streaming large data from server → client.

8. TLS / SSL / HTTPS – Security Basics

01:15:21.000SSL was the original protocol… now outdated due to vulnerabilities → replaced by TLS (current recommendation: TLS 1.3)

01:16:37.760HTTPS is basically HTTP + TLS encryption

Why Use HTTPS Everywhere?

  • Encrypts data in transit (confidentiality)
  • Prevents tampering (integrity)
  • Authenticates server identity via certificates
  • Modern browsers mark HTTP sites as “Not Secure”
